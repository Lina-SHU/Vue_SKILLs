# Vue.js 最佳實踐工作流程

請將此文件視為一份強制性的指導手冊。除非使用者明確提出異議，否則所有 Vue.js 相關任務都必須遵循以下工作流程。

## 核心原則
- **狀態的可預測性**：將原始狀態 (`ref`, `reactive`) 最小化，作為唯一的「真實來源」(Source of Truth)，並由此衍生出所有其他狀態 (`computed`)。
- **資料流的單向性**：嚴格遵守「Props 向下傳遞，Events 向上觸發」的單向資料流。
- **元件的專注性**：構建小而專注的元件，使其易於測試、重用與維護。
- **避免不必要的渲染**：善用 `computed` 屬性和 `watch` 監聽器，精準控制更新。
- **程式碼的可讀性**：編寫清晰、易於理解的程式碼，是最高優先級之一。

## 1. 動手前：確認架構與規劃 (必要步驟)

### 1.1. 確認技術棧
- **預設技術棧**：Vue 3 + Composition API + `<script setup>`。

### 1.2. 核心實踐指南 (必讀)
在執行任何任務前，必須閱讀並在過程中**持續應用**以下核心指南：
- `references/reactivity.md`
- `references/sfc.md`
- `references/component-data-flow.md`
- `references/composables.md`

### 1.3. 規劃元件邊界 (必要步驟)
在實作任何具備一定複雜度的功能前，必須先建立一份簡要的元件結構圖。

- **定義職責**：用一句話描述每個元件的單一職責。
- **組合層 (Composition Facade)**：將入口頁面 (Entry/Root) 和路由層級的視圖元件 (Route-level View) 視為「組合層」。它們的職責是**組織和串連**功能性元件，本身不應包含複雜的商業邏輯或 UI 實作。
- **定義契約**：為結構圖中的每個子元件，明確定義其 `Props` (接收的資料) 和 `Emits` (發出的事件)。
- **組織結構**：當需要新增多個元件時，優先採用基於功能的資料夾結構，例如 `components/<feature>/...`、`composables/use<Feature>.js`。

## 2. 實作中：應用基礎實踐 (必要步驟)

以下為基礎且必須掌握的實踐。請借助在 `1.2` 節中載入的核心指南，在每個任務中應用所有原則。

### 反應性 (Reactivity)
- **參考文件**: `references/reactivity.md`
- 最小化原始狀態 (`ref` / `reactive`)，並最大化地使用 `computed` 來衍生狀態。
- 僅在需要處理副作用 (side effects) 時才使用 `watch`。
- 嚴禁在模板 (`<template>`) 中放置複雜的表達式或耗時的計算。

### 單一檔案元件 (SFC)
- **參考文件**: `references/sfc.md`
- **結構順序**：嚴格遵循 `<script>` → `<template>` → `<style>` 的順序來組織區塊。
- **職責單一**：保持 SFC 的職責單一。一旦元件變得過於龐大或複雜，就應立即進行拆分。
- **宣告式模板**：保持模板的宣告性，將所有邏輯判斷與衍生狀態移至 `<script>` 區塊中。
- **模板安全**：遵循所有 Vue 模板的安全規則 (謹慎使用 `v-html`、為 `v-for` 指定 `key`、選擇正確的條件渲染指令 `v-if` vs `v-show`)。

### 元件拆分的客觀原則
當一個元件承擔了**一個以上的明確職責**時（例如：同時處理資料請求與 UI 呈現），就必須進行拆分。若滿足以下**任一條件**，就應立即拆分：

1.  **混合職責**：元件同時負責資料處理（狀態管理、流程控制）**和**複雜的畫面呈現邏輯。
2.  **區塊過多**：元件包含了 3 個或更多個不同的 UI 區塊（例如：表單、篩選器、列表、狀態列）。
3.  **可抽象化**：模板中的某個區塊是重複的，或可以被抽象化成一個可重用的部分（例如：表格的每一行、卡片、列表項目）。

#### 入口頁面與路由視圖的特殊規則：
- **保持簡潔**：入口頁面與路由層級的視圖元件應保持極度簡潔。它們主要用於：
    - 應用層級的版面配置 (Layout)。
    - 啟用全域 Provider (`provide`)。
    - 組合來自不同模組的功能性元件。
- **禁止直接實作**：當需要整合一個功能區塊時（例如 CRUD 列表），不應將完整的實作邏輯直接放在視圖元件中。至少應拆分為：
    - **功能容器元件** (Feature Container Component)
    - **輸入/表單元件** (Input/Form Component)
    - **列表/列表項元件** (List/Item Component)
    - **操作/狀態元件** (Actions/Status Component)
- **例外情況**：只有在開發極小規模、一次性的示範範例時，才允許將所有邏輯放在單一檔案中。如果選擇此作法，必須明確註明為何不需要拆分。

### 元件資料流
- **參考文件**: `references/component-data-flow.md`
- **預設模型**：嚴格使用「Props 向下傳遞，Events 向上觸發」作為主要的資料流模型。
- **v-model**：僅在需要雙向綁定的元件契約時才使用，且應理解其運作原理。
- **provide/inject**：僅在處理深層跨元件的依賴注入或共享上下文時才使用。

### Composables
- **參考文件**: `references/composables.md`
- **提取時機**：當一段邏輯需要被**重用**、具有**狀態**，或包含**複雜的副作用**時，應將其提取到 Composable 中。
- **職責分離**：將狀態管理和副作用邏輯從負責畫面呈現的元件中分離出來，放入 Composable。
